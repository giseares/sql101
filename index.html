Módulos de aprendizaje
Concepto
01 / 03
¿Qué es una Tabla?
Un contenedor específico para un solo tipo de entidad de la vida real. Organiza la información en filas (registros) y columnas (atributos).

Es el bloque de construcción básico de cualquier base de datos relacional.
Atributos
02 / 03
Columnas y Atributos
Representan las propiedades de lo que guardamos.

Regla de oro: todas las filas de una columna deben tener el mismo tipo de dato. En fecha_nacimiento no puede haber un nombre.
Registros
03 / 03
Filas y Registros
Cada fila es un objeto individual y único.

Si la tabla es de Doctores → cada fila es un doctor.
Si la tabla es de Citas → cada fila es una consulta médica concreta.
1 / 3
Flashcards · Módulo 01
¿Qué es una tabla en una base de datos relacional?
↻ clic para ver respuesta
Contenedor para un solo tipo de entidad, organizado en filas y columnas. El bloque base de cualquier base de datos relacional.
¿Cuál es la regla de oro de los atributos (columnas)?
↻ clic para ver respuesta
Todas las filas de una misma columna deben tener el mismo tipo de dato. No puedes mezclar, por ej., nombres con fechas en la misma columna.
¿Qué representa cada fila de una tabla?
↻ clic para ver respuesta
Un objeto individual y único. En una tabla de Doctores, cada fila es un doctor específico. En Citas, cada fila es una consulta médica concreta.
1:1
01 / 03
Relación Uno a Uno
Un registro de la Tabla A ↔ solo uno de la Tabla B.

Ej: Paciente y su Expediente Clínico.

Ambas tablas suelen compartir la misma Primary Key.
1:N · la más común
02 / 03
Relación Uno a Muchos
Un registro de A → muchos de B. Pero cada B pertenece a un solo A.

Ej: Hospital → Doctores.

La PK del Hospital se guarda como FK en la tabla Doctores.
N:N
03 / 03
Relación Muchos a Muchos
Múltiples registros de A se relacionan con múltiples de B.

Problema: un Paciente tiene muchos Doctores y viceversa.

Solución: tabla intermedia Appointments (Join Table).
1 / 3
Flashcards · Módulo 02
¿Cuál es la relación más común en bases de datos? Da un ejemplo.
↻ clic para ver respuesta
La relación 1:N (Uno a Muchos). Ejemplo: un Hospital tiene muchos Doctores, pero cada Doctor pertenece a un solo Hospital.
¿Cómo se implementa en la DB una relación 1:N?
↻ clic para ver respuesta
La Primary Key de la tabla "uno" se guarda como Foreign Key en la tabla "muchos". Ej: hospital_id en la tabla Doctores.
¿Por qué no podemos usar Foreign Keys directas en una relación N:N?
↻ clic para ver respuesta
Porque tendríamos que repetir datos infinitamente en ambas tablas. La solución es una tabla intermedia (Join Table) que conecta los IDs de ambas.
Primary Key
01 / 03
Llave Primaria
El DNI de cada registro. Garantiza que cada fila sea única.

• Nunca puede estar vacía (NOT NULL)
• No pueden existir dos filas con el mismo valor

Ej: paciente_id — dos "Juan Pérez" tendrán IDs distintos.
Foreign Key
02 / 03
Llave Foránea
Columna que apunta a la PK de otra tabla. Crea un vínculo entre tablas sin duplicar información.

Es el "ancla" para hacer un JOIN.

Ej: hospital_id en Doctores apunta a la tabla Hospitales.
Schema
03 / 03
Database Schema
La estructura completa de la base de datos.

• El schema = conjunto de tablas
• Cada tabla = conjunto de columnas
• Las FK definen las relaciones entre tablas

El schema es el "mapa" de toda la DB.
1 / 3
Flashcards · Módulo 03
¿Cuáles son las dos reglas absolutas de una Primary Key?
↻ clic para ver respuesta
1) No pueden existir dos filas con el mismo valor (unicidad). 2) Nunca puede estar vacía — siempre NOT NULL.
¿Para qué sirve una Foreign Key y por qué es clave para los JOINs?
↻ clic para ver respuesta
Permite relacionar información entre tablas sin duplicarla. Es el "ancla" del JOIN: le dice a SQL cómo conectar las filas de una tabla con las de otra.
¿Qué es un Database Schema?
↻ clic para ver respuesta
La estructura completa de la base de datos: el mapa de todas las tablas, sus columnas y las relaciones (FKs) entre ellas.
🥉 Bronze
01 / 03
Raw Tables — Capa Bronze
Punto de entrada. Los datos se guardan tal cual vienen de la fuente (APIs, CSVs, DBs transaccionales).

• Estado: inalterado. Errores y formatos extraños incluidos.
• Propósito: respaldo histórico. La "verdad original".
🥈 Silver
02 / 03
Business Objects — Capa Silver
Aquí ocurre la limpieza. Datos transformados para que tengan sentido para el negocio.

• Limpio, tipado, sin duplicados, fechas estandarizadas.
• Propósito: "versión única de la verdad".
• Tablas: Invoices, Items, TaxData.
🥇 Gold
03 / 03
Report Tables — Capa Gold
Capa final, optimizada para dashboards y reportes (Power BI, Tableau).

• Datos altamente agregados y procesados.
• Los cálculos ya están hechos: ventas_mensuales_por_region.
• Fácil de leer sin SQL avanzado.
1 / 3
Flashcards · Módulo 04
¿Por qué existe la capa Bronze si los datos están "sucios"?
↻ clic para ver respuesta
Sirve como respaldo histórico de la "verdad original". Si algo sale mal en el procesamiento posterior, siempre puedes volver a ella sin consultar la fuente externa.
¿Qué es la "versión única de la verdad" y en qué capa vive?
↻ clic para ver respuesta
Es el principio de que todos los analistas usan la misma fuente de datos limpia y estandarizada. Vive en la capa Silver (Business Objects).
¿Para qué sirve la capa Gold y quién la consume?
↻ clic para ver respuesta
Está optimizada para dashboards y directivos. Los cálculos ya están hechos, así las herramientas de visualización (Power BI, Tableau) no tienen que procesar millones de filas en cada carga.
SELECT
01 / 04
SELECT — ¿Qué columnas?
Define qué campos quieres ver. SELECT * trae todo (cuidado con raw grandes). AS renombra la columna en el resultado.
SELECT first_name, specialty AS esp
FROM doctors;
FROM
02 / 04
FROM — ¿De dónde?
Especifica la tabla o archivo fuente. En DuckDB puedes apuntar directo a un CSV sin importar nada.
FROM doctors
FROM 'data/raw/clientes.csv'
WHERE
03 / 04
WHERE — Filtrar filas
Solo verás registros que pasen esta condición. Operadores: = <> > < LIKE IN
WHERE age > 30
WHERE specialty = 'Psychology'
WHERE last_name LIKE 'H%'
COUNT · GROUP BY
04 / 04
COUNT + GROUP BY
COUNT(*) cuenta todas las filas (incluye nulls). COUNT(col) solo donde no es NULL — detecta datos faltantes. GROUP BY agrupa por categoría.
SELECT COUNT(*) AS total, specialty
FROM doctors
GROUP BY specialty
ORDER BY total DESC;
1 / 4
Flashcards · Módulo 05
¿Cuál es la diferencia entre COUNT(*) y COUNT(columna)?
↻ clic para ver respuesta
COUNT(*) cuenta todas las filas incluyendo nulos. COUNT(columna) solo cuenta donde esa columna no es NULL — ideal para detectar datos faltantes.
¿Qué ventaja ofrece DuckDB al leer datos raw con FROM?
↻ clic para ver respuesta
Puedes hacer FROM 'data/raw/archivo.csv' directamente sin importar el archivo a una base de datos primero. Acelera mucho el EDA inicial.
¿Para qué sirve ORDER BY col DESC?
↻ clic para ver respuesta
Para ordenar los resultados por una columna de mayor a menor (DESC) o menor a mayor (ASC, que es el default). Muy útil para ver los valores más altos primero.
