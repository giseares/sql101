Módulo 01

ABC de
Bases de Datos

01 —
Tabla
Contenedor para un solo tipo de entidad (ej: solo Doctores). Organiza la info en filas y columnas. Bloque base de cualquier DB relacional.
estructura
02 —
Atributo / Columna
Propiedad de la entidad. Regla de oro: todas las filas de una columna deben tener el mismo tipo de dato.
estructura
03 —
Fila / Registro
Cada fila es un objeto individual y único. Si la tabla es de Doctores, cada fila es un doctor específico.
estructura
04 —
Primary Key
El DNI del registro. Garantiza unicidad por fila. Nunca puede ser NULL. No pueden existir dos filas con el mismo valor.
llave
05 —
Foreign Key
Columna que apunta a la Primary Key de otra tabla. Permite relacionar info sin duplicarla. Es el "ancla" para hacer JOINs.
llave
06 —
Schema
La estructura completa de la base de datos: el conjunto de tablas, sus columnas y las relaciones entre ellas.
arquitectura
07 —
Relación 1:1
Un registro de A ↔ un registro de B. Ej: Paciente y su Expediente Clínico. Usualmente comparten Primary Key.
relación
08 —
Relación 1:N
Un registro de A → muchos de B. Ej: Hospital tiene muchos Doctores. La PK del Hospital se guarda como FK en Doctores.
relación · más común
09 —
Relación N:N
Muchos de A ↔ muchos de B. Se resuelve con una tabla intermedia (Join Table). Ej: Pacientes ↔ Doctores via Appointments.
relación
Módulo 02

ABC de
SQL

SELECT
¿Qué columnas quiero?
Define qué campos te interesan. Usa * para todo (cuidado con raw grandes). Alias con AS para renombrar columnas.
SELECT first_name, specialty AS especialidad
FROM
¿De dónde vienen los datos?
Especifica la tabla o archivo. En DuckDB puedes leer CSVs directamente sin importar primero.
FROM 'data/raw/clientes.csv'
WHERE
¿Qué filas me interesan?
Es el filtro. Operadores: = <> > < LIKE IN. Solo verás registros que pasen esta prueba.
WHERE age > 30 AND specialty = 'Cardio'
COUNT()
¿Cuántos registros hay?
COUNT(*) cuenta todo (incluye nulls). COUNT(col) solo donde no es NULL — útil para detectar datos faltantes.
SELECT COUNT(*), COUNT(email) FROM doctors
